## Purpose

`arcoris.dev/pool` is a small open-source Go library with a public issue
tracker, pull-request workflow, documentation set, benchmark/report layer, and
release-facing repository automation.

This project is intentionally technical, narrow in scope, and direct in review
style. People are allowed to disagree strongly about API shape, lifecycle
semantics, benchmark evidence, documentation quality, or repository direction.
They are not allowed to turn that disagreement into hostility, harassment, or
personal abuse.

This Code of Conduct exists to keep project spaces usable for contributors,
maintainers, and users who want clear technical discussion without personal
damage or avoidable noise.

## Expected behavior

People participating in this repository are expected to:

- engage professionally and respectfully;
- critique code, benchmarks, reports, proposals, and documentation without
  attacking the people behind them;
- keep discussion anchored to the repository, the current change, and the
  relevant evidence;
- read the linked project docs before escalating avoidable misunderstandings;
- be honest about uncertainty, missing evidence, compatibility risk, and trade-offs;
- accept maintainers' scope decisions, moderation decisions, and review
  boundaries even when they disagree;
- use the appropriate path for the issue at hand, including issue forms,
  pull-request review, and the separate security reporting path in [SECURITY.md](SECURITY.md);
- stop when asked to stop behavior that is making collaboration unworkable.

## Unacceptable behavior

The following behavior is unacceptable in repository spaces and related project
interactions:

- harassment, threats, intimidation, or sustained hostility;
- slurs, degrading language, or discriminatory behavior;
- sexualized language, imagery, or unwanted attention;
- insults, mockery, or demeaning remarks about another person's competence,
  identity, background, or motives;
- doxxing or sharing private information without explicit permission;
- trolling, flamebait, pile-ons, or repetitive antagonistic behavior;
- deliberate misrepresentation of another person's position, work, or intent;
- using technical bluntness as cover for abusive conduct;
- low-effort spam, issue flooding, or repetitive disruptive posting after
  prior direction from maintainers;
- publishing private moderation or conduct reports without permission;
- retaliating against someone for making a good-faith conduct or security report.

Direct technical criticism does not excuse abusive behavior.
Likewise, disagreement about project scope does not justify hostility.

## Scope

This Code of Conduct applies across project spaces, including:

- issues, pull requests, and review threads;
- discussions tied to repository work;
- documentation, benchmark, chart, and report discussions;
- maintainer-contributor communication directly related to the repository;
- public interactions where a person is clearly representing the project.

It applies to conduct, not to disagreement alone.

Maintainers may still close issues, reject proposals, request stronger
validation, or decline changes for scope, design, quality, evidence, or
maintenance reasons even when conduct is acceptable.

## Technical rigor is not misconduct

This repository has a high bar for correctness, clarity, and evidence.
The following are not conduct violations by themselves:

- rejecting a feature because it widens the package beyond its intended scope;
- asking for a minimal reproduction, stronger tests, or clearer validation;
- requiring benchmark evidence for performance claims;
- disagreeing sharply about lifecycle behavior, ownership rules, or repository
  architecture;
- closing issues or pull requests that do not fit the project;
- preferring focused changes over broad framework-style expansion.

These actions must still be carried out professionally.
The standard here is directness without contempt.

## Reporting

If you experience or witness unacceptable behavior:

1. prefer a non-public reporting path if the repository publishes one;
2. if the conduct also violates GitHub or another hosting platform's rules, you
   may also use that platform's reporting tools;
3. if no private conduct path is published, open only a minimal public request
   for a private route and do not post sensitive details publicly;
4. if the report is about a vulnerability rather than conduct, use
   [SECURITY.md](SECURITY.md) instead.

Please include, when possible:

- a link to the relevant thread, pull request, or discussion;
- usernames involved;
- a short factual description of what happened;
- timestamps, screenshots, or copied text when needed;
- whether the behavior is ongoing;
- any immediate safety or retaliation concern.

Do **not** open a detailed public issue for a conduct report unless there is no
safer option.

## Maintainer responsibilities

Maintainers are responsible for:

- protecting a professional working environment;
- enforcing this Code of Conduct fairly and consistently;
- separating technical disagreement from unacceptable behavior;
- reducing harm and de-escalating when practical;
- protecting reporter privacy as reasonably possible;
- documenting and communicating moderation outcomes at the level appropriate to
  the case.

Maintainers may remove, edit, hide, lock, or reject content that violates this
Code of Conduct. They may also temporarily or permanently restrict participation
when necessary to protect the project.

## Enforcement and moderation

When reviewing a conduct issue, maintainers should consider:

- severity of the behavior;
- whether the behavior is repeated;
- whether there is a pattern of escalation, disruption, or retaliation;
- whether the conduct makes collaboration unsafe or unworkable;
- whether the participant is willing to stop and correct the behavior.

Possible moderation responses include:

- a clarification or request to stop;
- a private warning;
- comment or content removal;
- thread locking or interaction limits;
- temporary restriction from project spaces;
- permanent ban from project spaces.

Not every case requires the same response.
The goal is to protect the project's collaboration environment, not to win an
argument.

## Project-specific notes

For `arcoris.dev/pool`, the following patterns are especially important:

- benchmark and report debates should stay about evidence, methodology, and
  interpretation, not about personal credibility;
- lifecycle and ownership discussions can be strict because the package
  contract is strict, but strictness is not a license for contempt;
- maintainers may say no to feature growth that turns this repository into a
  larger framework, resource manager, or policy surface;
- documentation review is part of product quality, so requests to synchronize
  docs with code are ordinary project standards, not hostility.

## Summary

Help keep this repository:

- technically rigorous;
- respectful in review;
- explicit in scope;
- and workable for contributors and users.

Strong technical standards and strong conduct standards are compatible.
This project expects both.
